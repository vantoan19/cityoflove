# Agent notes

## Sprite animation frames

When generating frames inside `chapter1/animations/**/pose*/` folders (e.g. `frame1.png`, `frame2.png`, …), treat them as **frames of a sprite animation loop**, not as independent illustrations.

Rules that apply to every frame in a pose folder:

- **Tiny per-frame deltas only.** Changes between adjacent frames are 1–2 px shifts or sub-degree tilts. Every frame must look essentially identical to frame 1 — no re-posing, no re-composing, no style drift.
- **Consistent direction.** The character must face the same way in every frame. Do not flip, mirror, or rotate the body/head between frames.
- **Consistent dimensions.** All frames in a single pose folder must share the exact same output resolution and aspect ratio. If one frame comes back at a different size (e.g. 896×1195 vs. 843×1264), regenerate it so dimensions match across the loop.
- **Consistent framing + scale.** Camera crop, character size within the frame, and ground/background position do not change between frames.
- **Consistent identity, outfit, palette, background.** Pull identity from `base.png` for the fox or `basee.png` for the bunny and hold it fixed across the loop.

### Workflow

1. **Generate frame 1 only** with `mode='generate'`, using `base.png` + pose reference as input images, and pin `aspect_ratio` (e.g. `2:3`).
2. **Generate frames 2..N with `mode='edit'`**, passing `frame1.png` as the sole `input_image_path_1` and a minimal prompt describing ONLY the tiny delta (e.g. "tilt the sunglasses 2° clockwise, change nothing else"). Same `aspect_ratio`.

Do NOT use `mode='generate'` for frames 2..N — even with frame 1 as a reference, generate mode re-draws from scratch and produces frames that look like different images rather than steps of one animation. Edit mode starts from the actual frame 1 pixels and only applies the delta, which is what a sprite loop needs.

### Accessory / prop consistency (sunglasses, ties, hats, tools, etc.)

Small rigid accessories are the **#1 source of sprite drift**. Even in `mode='edit'` with strict prompts, AI image models re-paint accessories on every pass, and small details (lens shape, frame thickness, tint, highlights, tie stripe count/width, hat brim, etc.) will not match across frames. This breaks the "same object moving" illusion that makes a sprite loop readable.

Rules:

- **Never ask the AI to rotate or move an accessory as the animation delta.** Rotating sunglasses, nodding a hat, wiggling a tie — the AI will redraw the accessory each frame and the shape/color/thickness will differ. Result: looks like a different pair of glasses every frame, not one pair tilting.
- **Pick deltas that are easier for the model to preserve:** whole-body 1-pixel bob, tail-tip shift, 1-px mouth-corner change, ear-tip flick, eyebrow raise. These are small, localized, and don't involve a rigid object that must match itself.
- **If the animation concept requires an accessory delta** (e.g. "adjusting sunglasses"), do it programmatically, not via AI: generate frame 1 with the accessory, then use PIL/ImageMagick to rotate/translate just the accessory region with the actual frame-1 pixels. That guarantees the accessory is the same object across frames, only transformed.
- **When describing an accessory in any prompt, lock it in full detail** — shape, outline weight, color, tint, highlights, visible features through it. Vague descriptions ("sunglasses") give the model license to reinterpret. But treat detailed descriptions as a last resort; they reduce drift but do not eliminate it.
- **Spot-check accessories explicitly after every regen** by reading 2-3 frames side-by-side. If the accessory differs, the loop is broken — do not ship it.
- **Do not deform a clothed region as the delta.** Torso stretches ("torso 1% taller"), shoulder shifts, hip twists force the AI to re-paint the clothing on that region, so uniforms/vests/stripes/badges will drift frame-to-frame. Keep deltas confined to bare body parts (ear tips, tail tip, mouth corner, eyebrow, standalone hand/finger) that don't intersect clothing. For a "breathing" feel, prefer ear-only or tail-only motion over torso scale changes.

## nano-banana MCP — resolution parameter only works in generate mode

The `zhongweili/nanobanana-mcp-server` (`mcp__nano-banana__generate_image`) has a wiring bug: the `resolution` parameter is **only forwarded to generate mode**. In edit mode (either `mode="edit"` or auto-detected when `input_image_path_*` is set), the `resolution` value is silently dropped and output is capped at ~1376×768 regardless of model tier — confirmed on both `nb2` (Gemini 3.1 Flash Image) and `pro` (Gemini 3 Pro Image). The response metadata shows `"media_resolution":"high"` instead of the requested `4k`/`2k`.

Evidence in the server source (`nanobanana_mcp_server/tools/generate_image.py`): `resolution` is passed to `selected_service.generate_images(...)` but **not** to `selected_service.edit_images(...)` or `edit_image_by_file_id(...)`.

### Workaround — force generate mode with the base as conditioning

```python
mcp__nano-banana__generate_image(
    prompt="...use the reference image as a strict visual template...",
    input_image_path_1="/path/to/base.png",   # still passed
    input_image_path_2="/path/to/prev_frame.png",  # optional, for chain
    mode="generate",         # KEY: explicit generate, not edit
    model_tier="nb2",        # or "pro"
    resolution="4k",         # now actually honored
    aspect_ratio="16:9",
    output_path="...",
)
```

**Prompt pattern shift.** In edit mode you'd write "edit this image to add X". In forced-generate mode with a reference, phrase as "use the reference image as a strict visual template to recreate the exact same scene pixel-for-pixel, then layer these additions on top". The input images act as multi-image visual conditioning rather than an edit target.

**When to use which mode** (project-wide, overrides the chapter-1 workflow note above when output resolution matters):

- Low-res (~1376×768) acceptable → use `mode="edit"` (simpler prompt surface, tighter identity anchor).
- Need true native 4K / base-matching resolution → force `mode="generate"` even when passing reference images. Accept that the model re-draws from scratch and has more style drift; compensate by writing a very prescriptive prompt about palette / silhouettes / stroke texture and, for sprite loops, chaining with the previous frame as a second reference.

**Verified:** city.png (5504×3072) → frame produced at 5504×3072 (7 MB) in forced-generate mode with `resolution="4k"` and `aspect_ratio="16:9"`.
